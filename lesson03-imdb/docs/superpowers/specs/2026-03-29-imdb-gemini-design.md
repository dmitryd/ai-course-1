# IMDB Gemini Movie Advisor Design

## Goal

Сделать локальный веб-сервис, который принимает название фильма, ищет наиболее популярный релевантный фильм по базе IMDB через RapidAPI, отправляет данные о найденном фильме в Gemini `gemini-3.1-flash-lite-preview` и возвращает пользователю краткую рекомендацию на русском языке: стоит смотреть фильм или нет, почему, с базовыми данными и ссылкой на IMDB.

## Scope

В первую версию входят:

- локальный сайт на `Next.js + TypeScript + App Router`;
- одно поле ввода для названия фильма;
- серверный API-роут, который скрывает ключи `RapidAPI` и `Gemini`;
- поиск наиболее подходящего и популярного фильма;
- генерация краткого ответа на русском языке через Gemini;
- отображение результата, ошибок и состояния загрузки.

В первую версию не входят:

- авторизация;
- история запросов;
- база данных;
- кэширование;
- ручной выбор фильма из списка кандидатов;
- дополнительные фильтры по году, жанру или типу.

## Architecture

Приложение состоит из двух слоёв:

1. Клиентская страница с формой поиска и карточкой результата.
2. Серверный API-роут внутри Next.js, который координирует вызовы внешних API.

Поток данных:

1. Пользователь вводит название фильма на русском или английском языке.
2. Клиент отправляет POST-запрос на внутренний роут приложения.
3. Сервер выполняет поиск кандидатов по IMDB через RapidAPI.
4. Сервер ранжирует результаты и выбирает лучший фильм.
5. Сервер запрашивает детальную информацию и рейтинг выбранного фильма.
6. Сервер формирует промпт и отправляет данные фильма в Gemini.
7. Сервер возвращает на клиент нормализованный ответ.
8. Клиент показывает карточку фильма, краткую рекомендацию и ссылку на IMDB.

## Main Components

### UI Page

Главная страница содержит:

- заголовок;
- короткое описание сервиса;
- одно поле `Название фильма`;
- кнопку отправки;
- индикатор загрузки;
- блок ошибки;
- карточку результата.

Интерфейс и все тексты отображаются на русском языке.

### API Route

Серверный роут принимает JSON с полем `query`, валидирует ввод, запускает поиск фильма, вызывает Gemini и возвращает готовый JSON для фронтенда.

Ответ API должен быть единым по форме, чтобы клиент не знал детали внешних сервисов.

### IMDB Search Module

Отвечает за:

- отправку запроса поиска в RapidAPI;
- приведение ответа к внутреннему формату;
- фильтрацию нерелевантных результатов;
- ранжирование кандидатов;
- получение деталей и рейтинга выбранного фильма.

### Gemini Recommendation Module

Отвечает за:

- подготовку безопасного краткого промпта;
- вызов Gemini `gemini-3.1-flash-lite-preview`;
- извлечение итогового текста рекомендации;
- возврат краткого ответа на русском языке в 2-3 предложениях.

## Movie Selection Strategy

Сервис не должен брать просто первый результат. Нужна детерминированная эвристика выбора наиболее подходящего популярного фильма.

Предлагаемая стратегия:

1. Нормализовать пользовательский ввод: обрезать пробелы, привести строку к нижнему регистру, убрать лишние повторяющиеся пробелы.
2. Получить список кандидатов из IMDB поиска.
3. Отфильтровать результаты, которые не похожи на полноценный фильм, если тип результата доступен.
4. Посчитать итоговый score для каждого кандидата.

Score строится из нескольких сигналов:

- сила совпадения названия с запросом;
- совпадение по основному заголовку и альтернативному заголовку;
- бонус за тип `movie`;
- бонус за высокую популярность;
- бонус за высокий рейтинг и большое число голосов, если такие поля доступны;
- штраф за документалки, ТВ-эпизоды, короткометражки и нерелевантные форматы, если это можно определить из ответа API.

Главный приоритет:

- сначала релевантность названия;
- затем тип `movie`;
- затем популярность и рейтинговые сигналы.

Это нужно, чтобы запрос вроде `Матрица` выбирал известный художественный фильм `The Matrix`, а не менее популярный побочный материал по теме.

Если точного совпадения нет, сервис берёт лучший близкий кандидат и показывает его как наиболее подходящий результат.

## Prompting Strategy

Gemini не ищет фильм самостоятельно и не должен заменять этап подбора фильма. Модель получает уже выбранный фильм и делает только краткую оценку на основе данных IMDB.

В промпт передаются:

- название фильма;
- оригинальное название, если есть;
- год;
- рейтинг IMDB;
- число голосов, если есть;
- жанры, если есть;
- краткое описание или сюжет, если есть;
- ссылка на IMDB.

Инструкция для модели:

- ответить только на русском языке;
- дать чёткий вывод, стоит смотреть или не стоит;
- дать 2-3 коротких предложения с причиной;
- опираться только на переданные данные;
- не придумывать факты;
- упомянуть рейтинг и общее впечатление от описания;
- вернуть короткий итог, пригодный для показа в UI.

## API Contract

### Request

`POST /api/recommend`

```json
{
  "query": "Матрица"
}
```

### Success Response

```json
{
  "movie": {
    "id": "tt0133093",
    "title": "The Matrix",
    "originalTitle": "The Matrix",
    "year": 1999,
    "rating": 8.7,
    "votes": 2100000,
    "plot": "A computer hacker learns about the true nature of reality.",
    "url": "https://www.imdb.com/title/tt0133093/"
  },
  "recommendation": {
    "verdict": "Стоит посмотреть",
    "summary": "Это один из самых сильных фантастических фильмов своего времени. Высокий рейтинг и культовый сюжет говорят в пользу просмотра."
  }
}
```

### Error Response

```json
{
  "error": {
    "code": "MOVIE_NOT_FOUND",
    "message": "Фильм не найден"
  }
}
```

Допустимые коды ошибок первой версии:

- `VALIDATION_ERROR`;
- `MOVIE_NOT_FOUND`;
- `UPSTREAM_IMDB_ERROR`;
- `UPSTREAM_GEMINI_ERROR`;
- `INTERNAL_ERROR`.

## Error Handling

Пользователь не должен видеть сырые ответы RapidAPI или Gemini.

Сценарии:

- Пустой ввод: вернуть ошибку валидации.
- Нет подходящего фильма: показать `Фильм не найден`.
- RapidAPI недоступен или вернул неожиданную структуру: показать `Сервис временно недоступен`.
- Gemini не ответил или вернул некорректный формат: показать `Не удалось получить рекомендацию`.
- Непредвиденная ошибка приложения: показать общий безопасный текст ошибки.

На сервере нужно логировать техническую ошибку для локальной отладки, но в UI возвращать только человекочитаемое сообщение.

## Environment Variables

Для локального запуска нужны:

- `RAPIDAPI_KEY`;
- `RAPIDAPI_HOST=imdb236.p.rapidapi.com`;
- `GEMINI_API_KEY`;
- `GEMINI_MODEL_ID=gemini-3.1-flash-lite-preview`.

Все переменные читаются только на сервере из `.env.local`.

## Testing Strategy

Минимальный набор проверок для первой версии:

- unit-тест для ранжирования фильмов;
- unit-тест для нормализации и валидации запроса;
- unit-тест для формирования промпта Gemini;
- integration-тест серверного обработчика с моками внешних API;
- smoke-проверка UI локально.

Особое внимание в тестах:

- `Матрица` должен выбирать известный фильм, а не нерелевантный контент;
- пустой ввод должен корректно валидироваться;
- ошибки внешних API должны давать ожидаемые коды и сообщения;
- ответ Gemini должен безопасно обрабатываться даже при неожиданной структуре.

## Deployment Fit

Архитектура первой версии должна без изменений переноситься на бесплатный Vercel:

- Next.js App Router поддерживается напрямую;
- API-роут будет работать как serverless function;
- переменные окружения можно будет перенести из `.env.local` в настройки проекта Vercel;
- постоянное хранилище не требуется.

## Success Criteria

Сервис считается готовым для локальной проверки, если:

- открывается локально в браузере;
- принимает одно текстовое название фильма;
- по запросу находит наиболее подходящий популярный фильм;
- выдаёт на русском 2-3 предложения с рекомендацией;
- показывает базовые данные фильма и ссылку на IMDB;
- корректно обрабатывает пустой ввод, отсутствие фильма и сбои внешних API.
